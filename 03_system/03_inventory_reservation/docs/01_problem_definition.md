# 1장: 문제 정의

## 핵심 질문

재고 예약 시스템은 결제 전에 재고를 임시로 확보하기 위한 장치다. 이 장에서는 실제 재고 차감과 임시 예약을 분리해야 하는 이유를 정리한다.

## 문제 상황

동일한 상품에 대해 여러 사용자가 동시에 결제 화면으로 진입할 수 있다. 시스템이 결제 완료 시점에만 재고를 확인하면, 결제 진행 중인 사용자가 확보했다고 생각한 재고가 다른 사용자에게 먼저 판매될 수 있다.

반대로 예약을 너무 오래 유지하면 실제로 판매 가능한 재고가 묶인다. 이 경우 화면상 품절처럼 보이지만 실제로는 판매되지 않은 재고가 남는 언더셀 상황이 생긴다.

## 핵심 요구사항

- 결제 전 재고를 임시로 확보할 수 있어야 한다.
- 결제 성공 시 예약을 실제 재고 차감으로 확정해야 한다.
- 결제 실패 또는 만료 시 예약을 해제해야 한다.
- 동시에 들어온 예약 요청이 같은 재고를 중복 점유하면 안 된다.
- 예약 상태와 실제 재고 원장이 장시간 불일치하면 안 된다.

## 이번 스터디의 단순화 조건

- 상품은 1개로 시작한다.
- 재고 위치는 1개로 시작한다.
- 요청당 예약 수량은 1개로 시작한다.
- 초기 재고는 100개로 둔다.
- 동시 요청은 500개까지 늘려본다.

## 실습 연결

- 레거시 구조는 Redis의 가용 수량과 MySQL 원장이 서로 어긋나는 상황을 확인한다.
- 개선 구조는 MySQL row lock으로 같은 재고 단위가 중복 예약되지 않는지 확인한다.
