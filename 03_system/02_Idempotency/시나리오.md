# 멱등성 시나리오

목표
- 같은 논리 주문 요청이 여러 번 들어와도 결제 승인이 중복으로 발생하지 않도록 한다.

시스템 구성
- 클라이언트가 `POST /api/payments`를 호출한다.
- 백엔드가 mock PG 계층을 호출한다.
- Redis가 멱등성 락과 최종 결과를 저장한다.

기대 동작
1. 첫 번째 유효 요청은 정상 처리된다.
2. 같은 키와 같은 payload로 들어온 동시 요청은 대기 후 같은 결과를 받는다.
3. 같은 키와 다른 payload로 들어온 동시 요청은 거부된다.
4. 새 주문을 시작하면 새 멱등성 키가 발급되어야 한다.

이 실습이 필요한 이유
- 결제 흐름은 중복 부작용이 특히 치명적인 영역이다.
- UI 재시도, 더블 클릭, 네트워크 재전송이 중복 승인을 만들면 안 된다.
